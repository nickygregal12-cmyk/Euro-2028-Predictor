# CLAUDE.md — Euro 2028 Predictor

Project conventions for Claude Code sessions. Read this before making changes.

## What this project is

A mobile-first Euro 2028 football predictor web app. Users predict every group match, build a knockout bracket, pick a champion, and compete in private leagues. Free-tier hosting (Netlify + Supabase).

**Source-of-truth documents (in repo root):**
- `docs/scoring-rules.md` — the complete scoring and tie-break specification. All scoring logic implements exactly this. Never invent or change a rule without updating this doc first.
- `docs/tournament-structure.md` — tournament facts, fixture skeleton, third-place ranking rules, and the round-of-16 allocation table. Bracket/structure logic implements exactly this. Never invent or change a rule (especially the R16 allocation table) without updating this doc first.
- `docs/build-todo.md` — the tiered build plan. Work top-to-bottom within the current tier. Tick items off when done. New ideas go in the parking lot, not the current tier.
- `docs/competition-structure.md` — how the Original Predictor and bonus games relate (the separation law), navigation, private-league scope, bonus-game specs, and build order. Never blur competition boundaries without updating this doc first.
- `docs/auth-plan.md` — how development proceeds without auth screens (dev auto-login) and exactly what auth work lands when. Never change the dev-user or auth approach without updating this doc first.

## Architecture rules (non-negotiable)

1. **Domain-first.** All tournament logic lives in `src/domain/tournament/` as pure functions: data in, data out. No React imports, no database access, no side effects. Follow the pattern in `calculateGroupTable.ts`.
2. **No business logic in components.** Pages and components call domain functions and render results. If a component is computing standings, scores, or bracket progression inline, that's wrong — extract it.
3. **Every domain function ships with unit tests in the same commit.** A domain function without tests is not done. Tests live in `tests/domain/`.
4. **The server is the authority on deadlines and locks.** Browser countdowns are cosmetic. Lock enforcement happens in Postgres functions / RLS, never client-side alone.
5. **Scoring must be deterministic and recalculable.** Given the same predictions and results, scoring always produces the same output, and the system can fully recalculate all points from source data. Corrections must never double-count.
6. **One source of truth for rules.** If code and `docs/scoring-rules.md` disagree, the doc wins — fix the code, or consciously update the doc first.
7. **Competitions are separate.** Original Predictor and bonus games are fully separate competitions — entries, predictions, and score events always carry their competition type; leagues belong to the Original Predictor only (`docs/competition-structure.md`).
8. **Dev user is invisible to the app.** No code outside the dev auto-login shim may special-case the dev user (`docs/auth-plan.md`).

## Git discipline

- Stage specific files — never `git add .`
- One logical change per commit (e.g. "add resolveGroupTies + tests"), not batches of unrelated work
- Run `npx vitest run` before committing; don't commit red tests
- Push to GitHub after each completed piece of work
- After a branch is merged to main, **delete the remote branch in the same session** — merged branches are history, not inventory (`git push origin --delete <branch>`)
- Never commit `.env` files (already gitignored)

## Database access & migrations

- Claude Code has **no database access** (no service-role key, no CLI) — every migration is applied manually by Nicky in the Supabase SQL editor. Write the SQL + the verification query; never attempt to apply, probe, or verify against a live DB.
- When Nicky confirms a migration applied **and verified** (real query output, not "Success"), the confirmation turn itself updates `docs/ops-pending-migrations.md` (and roadmap/build-todo/CLAUDE.md where relevant) — docs sync on confirmation is part of that same response, never a follow-up.

Phase 2 — **prediction autosave concurrency fixed** (client-side; no migration). The audit found real races in `PredictionsProvider.tsx`: no path enforced at-most-one-in-flight, so a debounced score save on a slow wire could overlap an immediate joker/second save for the **same row** and the older response could land last and clobber newer data; stale responses also set save-status unconditionally; and **Golden Boot swallowed errors** (`.catch(()=>{})` — no status, no retry). Fix = the house pattern (pure module + thin wiring): (1) **`src/domain/saveCoordinator.ts`** — a pure, unit-tested per-key state machine: at most ONE save in flight per key, new changes during flight coalesce into a single latest-pending state (intermediates never sent), a settling response is applied only if its `seq` matches the most recent issued request (**stale success/failure ignored** — never overwrites newer state), and failures auto-retry with geometric backoff up to a bound then surface `error` for manual retry. (2) **`src/app/providers/saveController.ts`** — a thin impure runner that executes the reducer's effects (fire the real Supabase save, run backoff timers, push per-key status to the UI), one instance for all keys. (3) **Provider wiring** — every autosave routes through it: match score **and** joker share ONE key per match (they're one DB row, so they must never race), plus ties, the bracket set, and golden boot. Local prediction state is never rolled back — only saved/unsaved status changes; the lock boundary is untouched (server still authoritative; a post-lock rejection now simply surfaces as a failed save). **Failure surfacing reuses existing language** — match cards / TieResolver / bracket already render `saving/saved/error`+retry; Golden Boot gained a non-blocking `Alert` + Retry on the Review Awards card (no new visual language). 18 new tests (13 reducer + 5 runner with fake timers), tsc + oxlint clean, 311 total green. **In-browser verification was blocked** by the known dev-Supabase live-Turnstile CAPTCHA (dev auto-login fails `no captcha_token found`, so no authed page loads locally — the exact thing prod-cutover step 9 fixes); the app was confirmed to **boot cleanly** with the changes (renders, no module/runtime errors), and the pure ordering + the runner's effect execution are both covered by tests. **Symptom (3) — server-side revision protection — is deliberately OUT of scope** (client-only session). Facts for that separate decision: the coordinator now prevents *client-originated* races within one tab, but the DB upserts remain unconditional last-write-wins (`match_predictions`/`predicted_progression`/`bonus_predictions` upsert full rows with a client `updated_at` the server ignores), so two devices/tabs editing the same entry, or any out-of-order arrival the client can't see, are still last-writer-wins. A true guarantee needs a **migration**: an optimistic-concurrency guard (compare-and-set on `updated_at` or a monotonic `version` column, rejecting an older write) — a schema + RPC change, hence its own session.

Phase 2 — **server-side write integrity** (migration `20260722120000_write_integrity.sql`, **applied + functionally verified on dev 2026-07-22**; the server half of last session's client-side autosave-concurrency work). Two guards in one append-only migration. **Part A — optimistic concurrency:** a monotonic `version` column on `match_predictions`/`predicted_progression`/`bonus_predictions` + a shared `enforce_write_version()` BEFORE UPDATE trigger — the client echoes the version it last read, a stale write is rejected with a DISTINCT non-retryable SQLSTATE `PT409` (so it's told apart from lock/joker/rate-limit, all `check_violation`), and version increments server-side. Phase-1 finding: the upsert path carries expected-version cleanly via BEFORE UPDATE + `DEFAULT 0` (NO before-insert force — that would corrupt `excluded.version` on upsert), so writes stay upserts (no RPC conversion). Composes with existing triggers (lock fires before version by name order → post-lock write still lock-rejected). **Client:** `saveCoordinator` gained a terminal, non-retryable `conflict` state (auto-retrying a conflict would fight the other writer); `saveController` classifies it via `isConflict`; services (`writeConflict.ts` + the three upsert wrappers) carry+return versions and map `PT409` → `VersionConflictError`; `PredictionsProvider` tracks per-row versions and resolves conflicts wholesale via `ConflictBanner` — a non-blocking Alert on the affected surface (group predictor / bracket / review) with **"Load latest"** (discard local for server) and **"Keep mine"** (re-issue against the freshly-read version); local state never changes without the user choosing. **Part B — `submit_entry()` bracket revalidation (SAFE PARTIAL, by decision):** redefined to add structural checks needing no bracket rebuild — reject orphan `r16` rows, require the exact 8-row winner-only shape, and scope every progression team to the entry's tournament — all STRICT additions that never reject a currently-valid entry. **The full identity/tree coherence rebuild** (rebuild the R16 draw from the predicted group order + best-thirds and verify each winner is a valid participant of its tie — catches swapped-winner / winner-not-a-participant) needs porting ~850 lines of bracket domain to byte-identical SQL and is **scoped to its own session** (an over-strict divergence would block valid submissions; recorded in `docs/build-todo.md`).
